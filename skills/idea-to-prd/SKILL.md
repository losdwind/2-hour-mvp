# 点子 → PRD

把一句话点子变成一份下游可直接消费的 PRD。下游消费者有两个：人（评审、开发），以及 prd-to-hifi 技能（按页面清单生成高保真交互稿）。所以页面清单的完整与准确比篇幅更重要。

## 第 1 步 · 问答收敛

用 AskUserQuestion 问，一次最多 3-4 题，别车轮战。必问的维度：

1. **目标用户与场景**。谁、在什么时候、解决什么痛。
2. **核心循环**。用户反复做的那个动作是什么（背单词 App 的核心循环是"学词-测验-奖励"）。产品成败在这一环，PRD 里要单独成节。
3. **MVP 边界**。第一版砍到什么程度。给用户几个从激进到保守的选项。
4. **差异化**。同类产品已存在时，这个凭什么赢。用户答不上来就如实写"待验证假设"。

用户已经给了详细描述时别重复问，只补缺口。

## 第 2 步 · 写 PRD

交付一份 markdown，文件名 `PRD-<产品名>.md`。结构固定如下（下游按此解析）：

```markdown
# <产品名> PRD v1.0

## 1. 一句话定位
## 2. 目标用户与核心场景
## 3. 核心循环          ← 用一段话 + 一个循环图（mermaid 或文字箭头）说清
## 4. 功能需求
   按模块列，每条含：功能描述、优先级(P0/P1/P2)、验收标准
## 5. 页面清单           ← 下游生成交互稿的直接输入，必须完整
   | 页面 id | 页面名 | 业务分组 | 核心元素 | 进入方式 |
   页面 id 用小写连字符（如 onboard-goal），后续交互稿的 DOM id 与截图文件名都沿用它
## 6. 用户旅程           ← 按业务逻辑串起页面清单里的 id，这个顺序就是未来演示视频的分镜顺序
## 7. 非目标（明确不做什么）
## 8. 关键指标
## 9. 开放问题与假设
```

写法要求：

- **页面清单宁全勿缺**。空状态、错误态、断签回归态这类"次要页面"在演示视频里往往是亮点，漏了下游补不回来。
- **每个功能的验收标准可观察可验证**（"3 分钟内完成引导"而设计不写"体验流畅"）。
- 核心循环里的每个动作都必须能在页面清单里找到承载页面，写完自查一遍。

## 第 3 步 · 交付与衔接

交付 PRD 文件后，主动告诉用户下一步可以用 prd-to-hifi 技能把第 5 节页面清单变成高保真交互稿，一句话带过即可。
